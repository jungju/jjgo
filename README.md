# jjgo

개발기, 하고 싶은 말, 개발 사례, 사진/영상까지 함께 기록하는 개인 개발 아카이브 프로젝트.

## 목적

이 프로젝트는 아래 내용을 한곳에 축적하고 공개하는 사이트다.

- 개발 기록(개발기, 시행착오, 회고)
- 하고 싶은 말(생각, 관점, 짧은 메모)
- 개발 사례(문제 해결 과정, 결과물, 배운 점)
- 관련 미디어(사진, 영상)

핵심 목표는 "빠르게 쓰고, 오래 남기고, 다시 찾기 쉽게 관리"하는 것이다.

## 개발 Rule

### 1. 콘텐츠 중심 구조

- 모든 원본 콘텐츠는 `content/`를 단일 출처(Single Source of Truth)로 사용한다.
- 페이지 코드는 `src/`에 두고, 원본 글/미디어는 `content/`에 분리한다.

### 2. 작성 형식

- 글은 Markdown(`.md`)을 기본 포맷으로 사용한다.
- 글은 `content/posts/<post-folder>/index.md` 구조를 사용한다.
- 드래프트 글 폴더는 `content/posts/draft-<post-folder>/` 접두 규칙을 사용한다.
- 글에 포함되는 이미지/영상 등 리소스는 같은 글 폴더 안에 함께 둔다.
- 메타 정보(태그/카테고리)는 `content/meta/`에서 관리한다.

### 3. 파일 네이밍 규칙

- 파일명은 소문자 kebab-case를 사용한다.
- 권장 패턴:
  - 글 폴더: `YYYY-MM-DD-title/` 또는 `draft-YYYY-MM-DD-title/`
  - 글 본문 파일: `index.md`
  - 사진: `YYYYMMDD-topic-seq.ext`
  - 영상: `YYYYMMDD-topic-seq.ext`

예시: `content/posts/2026-02-12-first-dev-log/index.md`, `20260212-build-diary-01.jpg`

### 4. 미디어 관리 규칙

- 사진/영상 원본은 `original/`에 저장한다.
- 서비스용 최적화 파일은 `optimized/` 또는 `encoded/`에 분리 저장한다.
- 썸네일은 `thumbnails/`에 저장한다.

### 5. 게시 품질 기준

- 발행 전에 제목, 날짜, 태그를 반드시 확인한다.
- 외부 링크는 깨진 링크가 없는지 확인한다.
- 개인 정보/민감 정보는 게시 전에 제거한다.

### 6. 변경 관리

- 큰 변경은 작은 단위 커밋으로 나눈다.
- "콘텐츠 변경"과 "코드 변경"은 가능하면 분리 커밋한다.
- 구조 변경 시 `README.md`와 `content/README.md`를 함께 갱신한다.

## 범위(MVP)

- 글/사진/영상을 폴더로 안정적으로 관리할 수 있어야 한다.
- 콘텐츠를 읽어 목록/상세 페이지로 연결할 수 있어야 한다.
- 최소한의 메타데이터(제목, 날짜, 태그) 기반으로 탐색 가능해야 한다.

## 콘텐츠 구조

콘텐츠 폴더 구조는 `content/README.md`를 기준으로 관리한다.
